Tokenomics

Understanding LLM API Pricing — input, output, caching, and context bloat

A 15-minute primer using Amazon Bedrock as the worked example.

What is a token?

A token is the atomic unit a language model reads and writes. It is not a word and it is not a character.

Tokenizers split text into sub-word pieces using algorithms like Byte-Pair Encoding (BPE) or SentencePiece. Common words become a single token; rare words and punctuation get split.

Rules of thumb (English)

  • ~4 characters per token
  • ~0.75 words per token
  • 1,000 tokens ≈ 750 words ≈ 1.5 pages
  • Code, JSON, and non-English text use more tokens per character

Why it matters

Every API charge is measured in tokens. Every context-window limit is measured in tokens. If you can’t estimate tokens, you can’t estimate cost or capacity.

Tokenization in action

The same sentence, broken into the tokens the model actually sees:

"The quick brown fox jumps over the lazy dog."

The quick brown fox jumps over the lazy dog.

10 tokens, 44 characters → ~4.4 chars/token. Leading spaces are part of the token.

Inflators

  • Code: useState → 2-3 tokens
  • JSON keys, brackets, quotes
  • Emoji and CJK characters
  • Long URLs and base64 blobs

Compactors

  • Common English words (1 token each)
  • Plain prose, simple punctuation
  • Numbers in small ranges

The billing model: input vs output

INPUT tokens

Everything you send to the model:

  • System prompt
  • User message
  • Conversation history
  • Tool/function definitions
  • Retrieved documents (RAG)
  • Attachments converted to text

OUTPUT tokens

Everything the model generates:

  • The assistant’s reply text
  • Tool call arguments
  • Streaming chunks (counted once)
  • Reasoning / thinking tokens (where billed)

Cost = (input_tokens / 1M) × input_rate + (output_tokens / 1M) × output_rate

Why output tokens cost ~5× more

Input is processed in parallel

The whole prompt is fed through the network in one big batched forward pass — the “prefill” phase. GPUs love this; it’s memory-bandwidth bound but highly parallel.

Output is generated one token at a time

Each output token is an autoregressive step: a full forward pass through the model just to produce the next token. The KV-cache helps, but you still pay per step.

Typical ratio

On most providers, output is priced 3×–5× higher than input.

Claude Sonnet 4.6 on Bedrock: $3 input / $15 output per 1M → 5×.

This is why summarization and translation can be cheap — and why a verbose model writing essays gets expensive fast.

Prompt caching: pay once, reuse cheap

Most prompts re-send the same prefix over and over: system prompt, tool schemas, loaded documents, conversation history.

Prompt caching lets the provider store the model’s internal state for that prefix and reuse it on the next request — skipping the prefill compute.

Two new line items

  • Cache write first time you mark a prefix — costs ~1.25× input rate
  • Cache read every subsequent hit — costs ~0.10× input rate (90% off)

Mechanics

  • You mark cache breakpoints in the prompt
  • Match must be exact, prefix-aligned (any change before the breakpoint busts the cache)
  • Bedrock TTL: ~5 minutes idle
  • Minimum ~1K tokens to be cacheable

Cache where the prompt is stable. Don’t cache where it changes every turn.

Example 1 — A simple Q&A

User asks one question, model answers. Claude Sonnet 4.6 on Bedrock.

Prompt: “Explain what a Kubernetes pod is in one paragraph.”

Reply: ~150 words → ~200 output tokens. Prompt incl. system + question ≈ 80 input tokens.

Input  :   80 tokens × $3  / 1,000,000 = $0.00024
Output :  200 tokens × $15 / 1,000,000 = $0.00300
                                    ────────────
Total per call                       ≈ $0.0032  (a third of a cent)

At this scale, you can run thousands of calls before pricing matters. The story changes when context gets large.

Example 2 — Long context, no cache

Doc-Q&A bot: a 50K-token policy document loaded into the prompt. User asks 100 questions over a session.

Per call:
  Input  : 50,000 tokens × $3  / 1,000,000 = $0.1500
  Output :    500 tokens × $15 / 1,000,000 = $0.0075
                                            ────────
                                            $0.1575

100 calls × $0.1575                         = $15.75

Notice: the 50K-token document is re-sent — and re-billed — on every single turn. The output is trivial; the input is doing the damage.

Example 3 — Same workload, with caching

Mark the 50K-token doc as cached. The first call writes the cache; the next 99 read from it.

Call 1 (cache write):
  50,000 × $3.75 / 1M = $0.1875
     500 × $15   / 1M = $0.0075
                       ───────
                       $0.1950

Calls 2–100 (cache read):
  50,000 × $0.30 / 1M = $0.0150
     500 × $15   / 1M = $0.0075
                       ───────
                       $0.0225 × 99 = $2.2275

Session total                       = $2.42

$15.75 → $2.42 = 85% savings — by changing one flag in the request, not the model or the prompt.

Where does context bloat come from?

The model has no memory between requests. To act “continuous,” the client re-sends everything, every turn. That means every loaded artifact is paid for again and again.

Skills

Each loaded skill injects instructions, examples, and metadata into the system prompt. Typical: 200–2,000 tokens each. Load 10 skills → 5K–20K baseline.

MCP servers

Every tool the server exposes adds a JSON schema, description, and parameter docs. Typical: 100–500 tokens per tool. A server with 30 tools = 3K–15K.

Past messages

The whole conversation history rides along on every turn — your prompts, the model’s replies, every tool call and tool result. Grows monotonically.

Add file reads, RAG retrievals, and screenshots-as-text and a “simple” agent session can sit at 80K input tokens by turn five.

How to keep the bill flat

Cache the stable prefix

System prompt + skills + MCP schemas + loaded docs rarely change within a session. Mark them as a cache breakpoint. The 90% read discount neutralises most context bloat.

Trim what you load

Only enable the skills and MCP servers you actually need for the task. Most agents never call 80% of the tools they advertise.

Compact history

Summarise older turns instead of carrying them verbatim. Many CLIs do this automatically near the context limit.

Right-size the model

Cheap models for routing/extraction, expensive ones for reasoning. Haiku-class models can be 5–20× cheaper than Opus-class for the same input.

Example 5 — The unused-skills tax

You loaded 28 skills. Over a 20-turn session you actually invoked 3. The other 25 rode along on every turn — paid for, never used.

Dead-weight skills :  25 skills × ~1,040 tokens avg  = 26,000 tokens
Per turn (no cache):  26,000 × $3 / 1M                = $0.078
20-turn session    :  20 × $0.078                     = $1.56 of pure waste

With caching       :  26,000 × $0.30 / 1M             = $0.0078
20-turn session    :  $0.1875 cache write + 19×$0.0078 = $0.34
                                                          (cache cuts the waste 78%)

Caching softens the blow but doesn’t make bloat free. You’re still paying to store tokens the model never uses, and you’re consuming context-window budget you could spend on the actual task.

Example 7 — Conversation history hoarding

40-turn debugging session. The user’s current question only depends on the last 3 turns.

No compaction

Full history carried verbatim: 60K tokens ride along on turn 41.

Cost (no cache): 60K × $3/1M = $0.18 just for history, on a single turn.

Cost across remaining 10 turns: $1.80+.

Compacted

Older 37 turns summarised to 4K bullet points; last 3 turns kept verbatim (4K).

History size: 8K tokens.

Cost (no cache): 8K × $3/1M = $0.024 per turn → $0.24 across 10 turns.

Compaction trades a small quality risk (lost detail in older turns) for a 7× reduction in carrying cost — and it frees window space for new work.

Example 9 — The cache-write tax on bloat

Caching is the obvious answer to bloat — but caching junk isn’t free.

You cache a 60K stable prefix containing:
   - 10K actually-needed system & skills      ← worth caching
   - 50K dead-weight skills + MCP schemas     ← waste

Cache write (Sonnet 4.6, 5-min TTL):
  60K × $3.75 / 1M = $0.225   ← what you paid
  10K × $3.75 / 1M = $0.0375    ← what you should have paid
                              ─────────────────
                              $0.1875 wasted on the cache write alone

Then on every cache miss (TTL expires, prefix changes), you pay that $0.1875 tax again. In a long, bursty session that’s easily 5–10 cache writes — $1+ wasted before any model output.

Lesson: trim before you cache. Caching a leaner prefix beats caching a fat one.

Example 10 — Cache invalidation cascade

The cache is prefix-aligned. Match has to be exact, byte-for-byte, up to the cache breakpoint. Change one token in the middle → everything after that point is uncached.

Stable prefix order

[ system        ]  ← stable
[ CLAUDE.md     ]  ← stable
[ skills        ]  ← occasionally edited
[ MCP schemas   ]  ← stable
[ history       ]  ← grows each turn
<cache breakpoint>

Toggle one skill → re-pay cache write for skills + MCP + history. ~50K re-cached.

Reordered

[ system        ]  ← stable
[ CLAUDE.md     ]  ← stable
[ MCP schemas   ]  ← stable
<cache breakpoint>
[ skills        ]  ← may change
[ history       ]  ← grows

Toggling a skill busts only the post-breakpoint slice. Save the cache for what’s truly stable.

Cost of getting this wrong: 50K × $3.75/1M = $0.19 per invalidation. Multiply by every time someone toggles a skill mid-session.

Example 11 — Bloat hurts quality, not just cost

Lost in the middle

Long-context studies repeatedly show models perform worst on information placed in the middle of a stuffed window. Bury the actual user request behind 50K of unused tool schemas and the model may genuinely miss it.

Slower time-to-first-token

Prefill scales with input length. A 10K prompt streams its first token in ~0.5s; a 150K prompt can take 5–15s before any output. Caching helps, but a cache miss on a fat prefix produces a long, awkward pause.

The double penalty

Bloated context costs you:

  • Money — every turn pays for unused tokens
  • Latency — longer prefill, slower TTFT
  • Quality — model attention dilutes; instructions get buried
  • Headroom — less window left for actual work

Example 12 — What good looks like

A minimum-viable-context discipline. Before each session, ask:

✅ Do

  • Enable only the skills you need for this task
  • Scope MCP servers per project, not globally
  • Drop large tool outputs after extracting what you need
  • Compact history once it stops being useful
  • Put stable content before the cache breakpoint
  • Run /context early; trim if >30% used before the first turn

❌ Avoid

  • “Just enable everything in case I need it”
  • Loading every MCP server on every project
  • Reading whole large files when grep would do
  • Pasting raw API responses when one field is what you wanted
  • Caching a fat prefix and calling it solved
  • Ignoring /context until you hit the limit

Lean baseline + caching + compaction = the bloated session’s cost cut by 80–95% with better model output.

Takeaways

  • Tokens are the unit of billing — and of every context-window limit. Estimate them or be surprised by them.
  • Output is ~5× input. Verbose models writing long answers cost more than they look.
  • Long context compounds. Every turn re-bills the entire prompt. A 50K-token document at turn 100 has been billed 100 times.
  • Caching is the single biggest lever. 90% off the input rate for stable prefixes. Often turns a $30 session into a $5 one.
  • Watch the bloat sources: skills, MCP servers, and accumulating history are the usual suspects. Load only what you use.
  • Use /context in Claude Code to see where your tokens are actually going — guesswork is expensive.
